VIVID AUTO DETAILS QUOTE INTEGRATION CONTRACT

Purpose
This file describes the current quote integration layer after the refactor that generalized the site away from a Google-specific form hookup.

Current operating mode
The live site is intentionally still running in legacy compatibility mode.
If a form is using the existing Google Apps Script URL through data-google-script, the JavaScript sends the legacy payload shape so the current website keeps behaving like the pre-refactor version.
The future-ready normalized payload is only used once the quote endpoint is changed away from the legacy Google Apps Script transport.


==================================================
1. WHERE QUOTE FORMS EXIST
==================================================

Homepage
- index.html
- form id: quoteForm
- variant: homepage

Services page
- services.html
- form id: quoteForm
- variant: services-page


==================================================
2. MARKUP-LEVEL CONFIGURATION
==================================================

Each quote form can provide these attributes:

- data-quote-endpoint
  The HTTP endpoint that should receive the quote request payload.

- data-quote-mode
  Fetch mode for the request.
  Current value: no-cors
  Expected future value for a custom backend: usually cors

- data-quote-variant
  Logical identifier for the form source.
  Current values:
  - homepage
  - services-page

Example
<form class="quote-form"
      data-quote-endpoint="https://your-endpoint.example.com/quotes"
      data-quote-mode="cors"
      data-quote-variant="homepage">


==================================================
3. SUBMISSION RULES
==================================================

The shared JavaScript does the following:
- intercepts submit on every .quote-form
- blocks duplicate submission
- checks the hidden honeypot field named website
- uses browser validity rules for required fields
- requires at least one checked service
- builds a normalized payload
- POSTs JSON to the configured endpoint
- shows a success state if submission succeeds

If data-quote-mode is no-cors
- the request is sent using fetch(..., { mode: "no-cors" })
- the frontend cannot inspect the response body
- success is assumed if no exception is thrown

If data-quote-mode is cors
- the request is sent as a normal JSON POST
- the frontend expects an HTTP success response


==================================================
4. PAYLOAD SHAPE
==================================================

The request body is JSON.

Legacy mode payload
When the form is still using the existing Google Apps Script transport, the payload is intentionally kept close to the original site behavior:
{
  firstName: string,
  lastName: string,
  email: string,
  phone: string,
  vehicle: string,
  services: string[],
  message: string,
  timestamp: string
}

Top-level fields
{
  firstName: string,
  lastName: string,
  fullName: string,
  email: string,
  phone: string,
  preferredContact: string,
  vehicle: string,
  vehicleYear: string,
  vehicleMake: string,
  vehicleModel: string,
  vehicleColor: string,
  services: string[],
  message: string,
  sourcePage: string,
  formVariant: string,
  timestamp: string,
  canonical: {
    contact: {
      firstName: string,
      lastName: string,
      fullName: string,
      email: string,
      phone: string,
      preferredContact: string
    },
    vehicle: {
      display: string,
      year: string,
      make: string,
      model: string,
      color: string
    },
    services: string[],
    message: string,
    sourcePage: string,
    formVariant: string,
    timestamp: string
  }
}

Why both flat fields and canonical fields exist
- Flat fields preserve compatibility with the current transition transport.
- canonical provides a cleaner target structure for a new CRM or API.


==================================================
5. FIELD MAPPING BY PAGE
==================================================

Homepage mapping
- firstName -> firstName
- lastName -> lastName
- email -> email
- phone -> phone
- vehicle -> vehicle
- message -> message
- checked service labels -> services[]

Services page mapping
- name -> fullName
- phone -> phone
- email -> email
- preferred-contact -> preferredContact
- vehicle-year -> vehicleYear
- vehicle-make -> vehicleMake
- vehicle-model -> vehicleModel
- vehicle-color -> vehicleColor
- combined year + make + model -> vehicle
- checked service labels -> services[]
- message -> message


==================================================
6. EXPECTED BACKEND BEHAVIOR
==================================================

Recommended minimal API contract for the replacement system

Endpoint
- POST /quotes

Request
- Content-Type: application/json
- Body: payload described above

Recommended successful response for cors mode
- HTTP 200 or 201
- JSON response optional

Example response
{
  "success": true,
  "quoteId": "abc123",
  "message": "Quote request received"
}


==================================================
7. CUTOVER PLAN
==================================================

To switch from the current transition transport to the new CRM system:

1. Replace data-quote-endpoint on each form with the new endpoint.
2. Change data-quote-mode from no-cors to cors if the new backend supports standard CORS.
3. Keep accepting the existing payload during migration.
4. Optionally remove legacy flat fields later if the backend uses canonical only.
5. Test both forms separately because they originate from different page layouts.


End of integration contract.
