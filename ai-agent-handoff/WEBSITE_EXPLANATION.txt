1. HIGH-LEVEL WEBSITE SUMMARY
==================================================

This website is a static multi-page marketing site for Vivid Auto Details.

Technology stack
- Plain HTML files
- One shared CSS file: style.css
- One shared JavaScript file: script.js
- Static image/video assets in images/
- No framework
- No build step
- No package.json
- No backend inside this repository

What the website does
- Markets the business and services
- Drives visitors to the external booking platform
- Shows service information, gallery, blog content, and FAQs
- Collects quote requests on the homepage through a Google Apps Script endpoint

What the website does NOT contain
- No local CRM backend
- No database
- No authentication
- No server-side routes
- No API layer in this repo

This means any CRM replacement or site-to-site integration will need to be connected from the frontend code to an external system or a new backend that you build elsewhere.

6. CURRENT QUOTE FORM: HOMEPAGE
==================================================

Location
- index.html
- Section id: contact
- Form id: quoteForm

Current endpoint location
The homepage form stores its submission endpoint in a generic HTML data attribute:

data-quote-endpoint="https://script.google.com/macros/s/AKfycbzmZHmiwRoL9ZHxpDMhfreazG55qm6PCuLMh6M5rWZgCSrxTrC2E2kT734tz6eoQyX8/exec"

The request mode is also explicit now:

data-quote-mode="no-cors"

Current homepage quote fields
- firstName
- lastName
- email
- phone
- vehicle
- services[] via checkboxes
- message
- hidden honeypot field: website

Homepage quote flow in JavaScript
The logic lives in script.js under the QUOTE FORM HANDLING section and now applies to all .quote-form instances.

Behavior summary
1. It finds the form by id="quoteForm".
2. It prevents double-submission with a formSubmitted flag.
3. It checks a hidden honeypot field named website.
4. It validates that email exists.
5. It validates that at least one service checkbox is selected.
6. It changes the submit button into a sending state.
7. It reads the generic endpoint URL from quoteForm.dataset.quoteEndpoint.
8. It reads the request mode from quoteForm.dataset.quoteMode.
9. It builds a JSON payload that includes both flat legacy fields and a canonical nested structure.
10. It POSTs to the endpoint using fetch().
11. If the mode is "no-cors", it assumes success if no exception is thrown.
12. If the mode is not "no-cors", it requires a successful HTTP response.
13. On success, it hides the form and shows the success message block.
14. On failure, it restores the button state and shows an alert.

Current JSON payload shape for the homepage form
{
  firstName: string,
  lastName: string,
   fullName: string,
  email: string,
  phone: string,
   preferredContact: string,
  vehicle: string,
   vehicleYear: string,
   vehicleMake: string,
   vehicleModel: string,
   vehicleColor: string,
  services: string[],
  message: string,
   sourcePage: string,
   formVariant: string,
   timestamp: ISO-8601 string,
   canonical: {
      contact: {...},
      vehicle: {...},
      services: string[],
      message: string,
      sourcePage: string,
      formVariant: string,
      timestamp: ISO-8601 string
   }
}

How selected services are built
- The script reads checked checkboxes with name="services"
- It uses the visible label text from .checkbox-text when available
- This means the submitted service strings are human-readable labels, not only raw checkbox values

Important integration implication
If you replace Google Apps Script with your own system, the easiest current hook is either:
- keep the existing homepage form markup and replace the endpoint + JS submission logic, or
- replace the current quote handling block in script.js entirely and send the same or updated payload to your own API / CRM service

==================================================
13. RECOMMENDED CRM REPLACEMENT FOCUS
==================================================

If the goal is to replace the current CRM / quote transport cleanly, the highest-value tasks are:

1. Standardize quote forms
   Decide whether the homepage and services page should share one schema or remain separate.

2. Replace the Google Apps Script transport
   Update the homepage quote submission to point to your new system.

3. Define a canonical payload
   Make sure both quote entry points, if both remain, map into the same backend structure.

4. Decide how booking and quoting should coexist
   Right now "Book Now" routes to Fieldd while quote requests route elsewhere.
   If your new system replaces both, those links and form handlers should be aligned together.

5. Preserve shared site behavior when editing script.js
   Many unrelated visual and navigation features live in that file.
